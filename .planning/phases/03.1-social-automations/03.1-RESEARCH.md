## Meta Graph API — Comments & Replies

### List Comments on a Media Object
```
GET /{ig-media-id}/comments
  ?fields=id,username,text,timestamp,like_count,replies
  &access_token={token}
```

### Reply to a Comment
```
POST /{ig-comment-id}/replies
  ?message={reply_text}
  &access_token={token}
```

Response: `{ "id": "{reply-comment-id}" }`

### Hide/Delete a Comment
```
DELETE /{ig-comment-id}
  ?access_token={token}
```

### Get Comment by ID
```
GET /{ig-comment-id}
  ?fields=id,username,text,timestamp,like_count
  &access_token={token}
```

## Instagram Messaging API (DMs)

### Limitations
- Instagram Graph API does **NOT** support sending DMs directly via the standard Graph API
- DMs require the **Instagram Messaging API** which is part of the Meta Business Suite / Messenger Platform
- Requires: `instagram_manage_messages` permission (restricted, requires app review)
- Alternative: Use Facebook Messenger API with Instagram integration

### Workaround for v1
Since direct DM sending is restricted, the initial implementation will:
1. Implement the DM action in the data model and execution engine
2. For Instagram, DM action will be logged as "pending — requires Messaging API permissions"
3. Comment reply action is fully functional via Graph API
4. Future phase can add Messaging API integration once app review is complete

## Webhook Events for Comments

### Instagram Webhook Subscription
Subscribe to `instagram` object with field: `mentions` or use the Instagram Graph API webhook for comments.

Actually, Meta webhooks for Instagram comments work via the **Instagram Graph API webhook**:
- Object: `instagram`
- Fields: `mentions` (for mentions), but comments require polling or the Instagram Basic Display API

### Real Comment Webhooks
Meta does NOT provide real-time webhooks for all comments. Options:
1. **Polling approach**: Cron job polls `/{ig-user-id}/media?fields=comments` periodically
2. **Mentions webhook**: Only get webhooks for mentions (@username)
3. **Instagram Graph API webhook**: Available for some partners, requires special permissions

### Recommended Approach for v1
Use a **hybrid approach**:
1. Webhook infrastructure stays in place for future platform support (TikTok, YouTube may have comment webhooks)
2. For Instagram comments: implement a **polling-based comment watcher** as a Vercel cron job
3. Cron runs every 1-5 minutes, fetches recent comments on tracked posts
4. New comments are matched against automation triggers
5. This is the same pattern ManyChat and similar tools use for Instagram

## Automation Engine Architecture

### Data Model

```typescript
// automations table
interface Automation {
  id: string;
  userId: string;
  socialAccountId: string; // which connected account
  name: string;
  platform: "instagram" | "tiktok" | "youtube" | "x";
  triggerType: "comment_keyword";
  triggerConfig: {
    keywords: string[]; // ["EU QUERO", "quero"]
    matchMode: "exact" | "contains"; // default: contains
    caseSensitive: boolean; // default: false
  };
  actions: AutomationAction[];
  scope: {
    posts: "all" | "specific";
    postIds?: string[]; // external post IDs if specific
  };
  isActive: boolean;
  createdAt: Date;
  updatedAt: Date;
}

// automation_actions table (or JSONB in automations)
interface AutomationAction {
  id: string;
  automationId: string;
  type: "reply_comment" | "send_dm";
  config: {
    messages: string[]; // randomized responses
  };
  order: number; // execution order
  isActive: boolean;
}

// automation_logs table
interface AutomationLog {
  id: string;
  automationId: string;
  triggerEvent: {
    type: string;
    payload: Record<string, unknown>;
  };
  actionResults: Array<{
    actionId: string;
    status: "success" | "failed" | "skipped";
    output?: string;
    error?: string;
  }>;
  status: "success" | "partial" | "failed";
  executedAt: Date;
}

// comment_watch_state table (for polling)
interface CommentWatchState {
  id: string;
  socialAccountId: string;
  postId: string; // external post ID
  lastCheckedAt: Date;
  lastCommentId?: string; // cursor for pagination
}
```

### Execution Flow

```
1. Cron triggers (every 1-5 min)
2. Find all active automations with comment_keyword trigger
3. For each automation, get target posts (all or specific)
4. For each post, fetch comments since last check
5. For each new comment:
   a. Check cooldown (has this user triggered this automation recently?)
   b. Match keywords against comment text
   c. If match: execute actions in order
   d. Log execution
6. Update watch state (lastCheckedAt, lastCommentId)
```

### Keyword Matching

```typescript
function matchesKeyword(commentText: string, keywords: string[], matchMode: "exact" | "contains", caseSensitive: boolean): boolean {
  const text = caseSensitive ? commentText : commentText.toLowerCase();
  return keywords.some(keyword => {
    const kw = caseSensitive ? keyword : keyword.toLowerCase();
    if (matchMode === "exact") {
      return text === kw;
    }
    return text.includes(kw);
  });
}
```

### Random Response Selection

```typescript
function selectRandomResponse(messages: string[]): string {
  if (messages.length === 0) return "";
  if (messages.length === 1) return messages[0];
  const index = Math.floor(Math.random() * messages.length);
  return messages[index];
}
```

## Rate Limits & Safety

### Instagram Graph API Rate Limits
- 200 calls/hour per user for comments
- 100 API-published posts per 24 hours
- Comment replies count toward the 200 calls/hour limit

### Safety Measures
1. **Deduplication**: Store processed comment IDs to prevent re-processing
2. **Cooldown**: 1-hour cooldown per (automation, commenter) pair
3. **Rate limit tracking**: Track API calls per hour, pause if approaching limit
4. **Max executions**: Cap at 100 actions/hour per account

## Common Pitfalls

1. **DM sending without permissions**: Will fail at runtime. Must handle gracefully and show user-friendly error.
2. **Comment polling frequency**: Too frequent = rate limit hits. Too infrequent = delayed responses. Start with 2-minute intervals.
3. **Duplicate comment processing**: Without deduplication, same comment triggers automation multiple times. Must store processed comment IDs.
4. **Case sensitivity**: Users expect "EU QUERO" to match "eu quero". Default to case-insensitive.
5. **Emoji and special characters**: Keyword matching must handle Unicode correctly. Use normalized strings.
6. **Long-running cron on Vercel**: Vercel serverless functions have timeout limits (max 300s on Pro). Comment polling must complete within this window.
