<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> API | Untrusted input for link creation and redirect |
| API -> Database | Internal trusted boundary |
| Redirect -> External URL | User is redirected to potentially malicious URLs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2.1-01 | Spoofing | Custom slug creation | mitigate | Validate slug format (alphanumeric/hyphen/underscore only), block reserved words, enforce uniqueness |
| T-2.1-02 | Tampering | Click analytics | accept | Click data is best-effort; IP can be spoofed via VPN/proxy. No business-critical decisions depend on exact click counts |
| T-2.1-03 | Repudiation | Link creation | mitigate | Associate userId with created links; anonymous links accepted but logged with IP |
| T-2.1-04 | Information Disclosure | Password-protected links | mitigate | Hash passwords with bcrypt before storage; never return password in API responses |
| T-2.1-05 | Denial of Service | Redirect endpoint | mitigate | Rate limit 100 req/min per IP; index on slug for fast lookups |
| T-2.1-06 | Elevation of Privilege | Link management | mitigate | All link mutations require authenticated session matching link.userId |
| T-2.1-07 | Redirect to malicious URL | Redirect endpoint | mitigate | Validate originalUrl is a valid URL; consider URL reputation check in future phase |
</threat_model>
